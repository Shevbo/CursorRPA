# Shectory Portal: Control Panel Roadmap

Дата актуализации: 2026-03-26  
Статус: рабочий план реализации (single source of truth для следующих итераций)

**Контекст репозитория:** Shectory Portal — часть **монолита CursorRPA** (`CursorRPA/shectory-portal/` в Git), вместе с docs, scripts и `services/telegram-bridge/`. Публичный Web UI разворачивается с этого дерева (или из зеркала деплоя); см. `docs/shectory-projects-registry.md`.

**Навигация:** разделы **0–7** — содержание и детализация по доменам; **раздел 8** — **рекомендуемый порядок фаз** (от простого к сложному, бэклог вперёд); **раздел 9** — критерии готовности.

---

## 0) Резюме: сервисы CursorRPA для прикладных проектов

Ниже — целевая модель **платформенных** возможностей монолита: что даётся всем прикладным проектам в `~/workspaces` через панель, API и (где уместно) Telegram. Детали реализации и перенос из komissionka — в разделах 3–6; текущий код — в разделе 1.

1. **Пользователи, роли и доступы + модель аутентификации**  
   Каталог пользователей, аутентификация (эволюция от админ-токена к полноценным сессиям при необходимости), **RBAC в разрезе проекта**: кто видит бэклог, деплой, ключи ИИ, тесты и т.д. См. раздел 3.1.

2. **Управление бэклогом (основной use case)**  
   Фиксируем хотелки, **группируем в спринты**, переводим в работу. У каждой фичи/бага — **отдельный чат с агентом** (не одна бесконечная лента): сохраняется нить смысла, можно **параллелить** много фич; админу проще понять, **какой контекст сейчас у агента** по конкретной задаче. См. раздел 4.

3. **Тест-кейсы**  
   Практическая опора для **автоматизации** проверки фич; в перспективе явная **сквозная связка с бэклогом** (ссылки, статусы, трассировка «задача → сценарии → прогоны»). См. раздел 5.

4. **Деплой, среды, версионирование**  
   **Единые правила** выката, разворот сред, учёт **версий модулей** прикладных проектов (метаданные в портале + согласованность с git/артефактами). См. раздел 6.

5. **Единый сервис доступа к моделям ИИ**  
   Проекты ходят в LLM через **управляемые ключи** (коммерческие и бесплатные провайдеры): учёт **расхода токенов**, **очередь** запросов, обработка ошибок, аккуратный диалог с моделями и передача результатов вверх по стеку. В перспективе — **тарификация / квоты** использования для приложений. См. раздел 3.2.

6. **Интерфейс через Telegram**  
   Единый **стиль** управления в чате: согласованные команды, кнопки, системные сообщения; тот же «язык» управления, что и в Web-панели там, где это разумно (`services/telegram-bridge`, развитие ботов под проекты).

7. **Базовые сервисы (горизонт)**  
   Централизованное управление общими утилитами: файлообменник, SMTP, SMS-шлюз, (видео)звонки, обработка изображений и видео и т.п. — как **подключаемые** сервисы платформы, не дублируемые в каждом прикладном репозитории.

8. **Голосовой интерфейс с агентом (горизонт)**  
   Возможность **голосового ввода** с последующей **структуризацией речи** в понятные промпты и **валидацией** перед отправкой в Cursor Agent / LLM — отдельная итерация продукта, после стабилизации ядра п. 1–6.

## 3) Общие функции монолита CursorRPA для прикладных проектов (панель управления)

Два **платформенных** направления — не привязаны к одному продукту; подключаются из **панели управления** и из API с привязкой к `projectId` / `slug`, где уместно.

### 3.1. Каталог пользователей, аутентификация и ролевой доступ (RBAC)

**Цель:** единый модуль в репозитории CursorRPA (данные + API + UI), чтобы прикладные проекты в портале имели предсказуемую модель «кто что может делать» в control plane.

**Состав (черновик границ):**

- **Каталог пользователей:** учётные записи (email/login, профиль, статус), опционально привязка к внешнему IdP позже.
- **Аутентификация:** эволюция от текущего `ADMIN_TOKEN` + cookie; целевой вариант — нормальные сессии или SSO-готовая схема без привязки к одному вендору в документе.
- **Авторизация:** роли и разрешения **в разрезе проекта** (и при необходимости глобальные роли платформы): например доступ к backlog, deploy, bot, файлам, настройкам секретов, приглашениям.
- **UI в панели управления:** раздел «Участники / доступ» на `/projects/[slug]/control` (или эквивалент), матрица ролей, аудит последних изменений прав (минимально — лог в БД).

**Принципы:**

- Не ломать существующие сценарии в первой итерации: `adminAuthOk` остаётся для полного доступа; RBAC накладывается как более узкий режим для дополнительных пользователей.
- Секреты пользователей (пароли) — только хэш + стандартные практики; ключи внешних API — через модуль из п. 3.2, не в открытом виде в RBAC-таблицах.

**Этапы (высокоуровнево):**

- **B0:** модель Prisma (`User`, `ProjectMembership`, `Role` / enum прав), миграции.
- **B1:** API: приглашение, назначение роли, отзыв; проверка прав в middleware/helper для роутов `/api/project/*`.
- **B2:** UI панели + согласование с unified control panel (раздел 2).
- **B3:** (опционально) синхронизация с workspace/git (отдельное решение, не блокер).

### 3.2. Сервис доступа к сторонним моделям ИИ (перенос опыта из komissionka)

**Цель:** в монолите CursorRPA — отдельный **AI access / gateway** слой: вызовы не только через Cursor Agent, но и через **другие модели** (OpenAI-совместимые API и аналоги), с безопасной работой с ключами и единой точкой для портала.

**Что перенести по смыслу из komissionka** (без копипасты кода в этом документе):

- **Прокси / edge API keys:** серверная сторона хранит ключи, клиент и прикладной код получают только capability-токены или вызовы через backend; учесть rate limit и учёт использования.
- **Конфигурация моделей:** список провайдеров, моделей, дефолты на уровне проекта или платформы.
- **Опыт эксплуатации:** логирование запросов (без сырого промпта в прод-логах при политике приватности — настраиваемо), обработка ошибок провайдера, таймауты.

**Интеграция с панелью:**

- Раздел **«Модели / ключи»** в control panel проекта: привязка к env, ротация, кто из RBAC (п. 3.1) может редактировать.
- Потребители внутри портала: **Generate Prompt** (этап A3), чат `/api/agent/chat` (где уместно маршрутизировать на gateway), будущие сценарии прикладных проектов.

**Этапы (высокоуровнево):**

- **C0:** контракт внутреннего API `POST /api/ai/complete` (или несколько эндпоинтов), конфиг из БД + env fallback.
- **C1:** хранение секретов (шифрование at rest — по мере необходимости; минимум — env + отдельная таблица с ограничением доступа).
- **C2:** UI в панели + связка с RBAC (только роли с правом `ai.keys.manage` или аналог).
- **C3:** вынос/адаптация проверенных паттернов из кодовой базы komissionka (рефактор под multi-project).

**Зависимости:** в **разделе 8** RBAC (B) и AI gateway (C) вынесены **после** стабилизации бэклога и основных панелей: до этого везде достаточно `adminAuthOk`. C0–C1 допускается **параллельно** фазам P5–P6 при отдельном исполнителе; B включается, когда появляются реальные не-админ пользователи.

## 5) Детальный план: тест-кейсы (komissionka → Shectory Portal)

**Когда внедрять:** фазы **P5** в **разделе 8** (после стабильного бэклога и по желанию после A3).

Источник истины в **komissionka**: `prisma/schema.prisma` (`test_modules`, `test_cases`, `test_runs`), API под `src/app/api/admin/test-cases/*`, `test-modules`, UI `AdminTestCatalogTab.tsx`, описание домена — `docs/manual/test-catalog-architecture.md`.

**Текущий портал:** модели `TestModule` / `TestCase` с привязкой к `projectId`, роуты `GET|POST /api/project/tests` и при необходимости точечный PATCH по id — **упрощённый** набор полей, без глобального номера кейса, без прогонов (`test_runs`), без богатых метаданных и запуска из API.

### Этап D0. Аудит и контракт parity

- Сверить поля `test_cases` / `test_modules` komissionka с целевой схемой портала (таблица маппинга в комментарии к миграции или в `docs/`).
- Зафиксировать семантику:
  - `number` в komissionka **глобально уникален**; в multi-project портале — **уникален в пределах `projectId`** (`@@unique([projectId, number])`).
  - `module_id` в komissionka — строковый id модуля; в портале уже `moduleId` FK → сохранить возможность **стабильных строковых slug модулей** (как в komissionka) или маппинг `id` → `name` при миграции.
- Словари: `kind` (`manual-guided`, `semi-automatic`, `automatic`), `scope` (`ui`, `api`, `agent`, `telegram`, `db`, `infra`), статусы прогона `test_runs.status`.

### Этап D1. Расширение Prisma в портале

Добавить в `TestCase` (nullable где нужно для non-breaking):

- `number` (Int), уникальный в паре с `projectId`
- `promptTemplate`, `parameters` (Json), `expectedResult` (Json)
- `tags` (массив строк или Json — по согласованию с PostgreSQL/Prisma)
- `enabled` (Boolean, default true)
- `uiPages`, `apiEndpoints`, `codeRefs`, `dbEntities` (массивы строк)
- `specEnrichedByAi`, `specEnrichedAt`, `specEnrichedModel`
- при необходимости расширить `TestModule`: явный **строковый ключ модуля** (`key` / `slug`), если отличается от cuid `id`

Индексы: `[projectId, number]`, `[projectId, moduleId]`, `[projectId, enabled]`.

### Этап D2. Модель прогонов `TestRun`

- Новая таблица `TestRun` по аналогии с `test_runs` komissionka: связь с `TestCase`, `runNumber`, `startedAt` / `finishedAt`, `status`, `runner`, `requestContext`, `agentLogId`, `conversationLog`, `steps`, `comparisonResult`, `diagnostics` (Json по необходимости).
- Индексы: `[testCaseId]`, `[status]`, `[startedAt]`.

### Этап D3. API parity (project-scoped)

Расширить/добавить под префиксом `/api/project/tests` (или вложенные пути):

- `GET` — фильтры (модуль, scope, kind, enabled, тег), пагинация, сортировка (в т.ч. по `number`).
- `POST` — создание; опция **копирования** существующего кейса (как в komissionka: next `number` в проекте).
- `PATCH` / `DELETE` по id — полное обновление полей, мягкое отключение через `enabled`.
- `POST .../[id]/run` — перенос логики из `komissionka/src/app/api/admin/test-cases/[id]/run/route.ts` (адаптация под `projectId`, agent pipeline портала, пути workspace).
- CRUD **модулей** проекта (аналог `api/admin/test-modules`): список, создание, `active` flag.

Все вызовы — `adminAuthOk` (позже RBAC п. 3.1).

### Этап D4. UI в unified control panel

- Компонент `TestCasesPanel`: таблица + детальная карточка кейса, редактирование Json-полей (с валидацией), фильтры как в komissionka-админке.
- Отображение последнего прогона / истории `TestRun` (кратко).
- Кнопка «Запустить» → вызов `.../run` и вывод статуса/ошибок в едином блоке результатов (как в разделе 2).

### Этап D5. Экспорт и отчёты (опционально)

- Портировать по смыслу `src/lib/test-run-export.ts` (markdown/JSON отчёт по прогонам) в `shectory-portal/src/lib/`, путь вывода — относительно `project.workspacePath` или артефакт в UI.

### Этап D6. Миграция данных из komissionka

- Скрипт `scripts/migrate-test-catalog-from-komissionka.ts`:
  - чтение из БД komissionka (отдельный `DATABASE_URL` или dump);
  - создание/сопоставление `TestModule` в проекте `slug=komissionka`;
  - перенос `test_cases` с пересчётом уникальности `number` в scope проекта (если в komissionka один проект — прямой перенос номеров);
  - перенос `test_runs` с сохранением связи на новые id кейсов (маппинг uuid → uuid).
- Документировать ограничения (расхождение версий Prisma, ручной dry-run).

## 6) Детальный план: деплой (komissionka → Shectory Portal)

**Когда внедрять:** фаза **P6** в **разделе 8** (после тестов или параллельно хвосту P5 только на уровне E0–E1, если есть отдельный исполнитель — см. 8.3).

Источник в **komissionka**: `deploy_environments`, `deploy_queue`, `deploy_log` в `prisma/schema.prisma`; API `src/app/api/deploy/*` (environments, environments/[id], queue, queue/[id], status, copy); фоновый обработчик `scripts/deploy-worker.ts`.

**Текущий портал:** модель `DeployEnvironment` (имя, ветка, статус, `targetHost`, `directory`, `isProd`) без портов, без имени БД, без **очереди** и **журнала** операций; API `GET|POST /api/project/deploy`, `PATCH /api/project/deploy/[id]` — минимум.

### Этап E0. Аудит и целевая модель

- Зафиксировать, какие поля komissionka обязательны для **parity**: `port_app`, `port_agent`, `port_bot`, `db_name`, уникальность имени среды (в komissionka имя глобально уникально; в портале — **`@@unique([projectId, name])`**).
- Операции очереди: какие `operation` значения используются в komissionka (`deploy`, `copy`, …) — перенести справочник в константы портала.
- Решить, где крутится worker: **тот же хост**, что и портал (user systemd unit), или отдельный процесс с доступом к `workspacePath` и скриптам деплоя.

### Этап E1. Расширение Prisma `DeployEnvironment`

Добавить поля (nullable для старых строк):

- `portApp`, `portAgent`, `portBot` (Int, optional)
- `dbName` (String, optional)
- при необходимости `metadata` (Json) для специфичных полей без раздувания схемы

Сохранить `targetHost` / `directory` портала как основной способ адресации для **не-komissionka** проектов.

### Этап E2. Очередь и журнал

- Модели `DeployQueueItem` и `DeployLogEntry` по смыслу komissionka, с **обязательной** привязкой `projectId` (и `environmentId`):
  - очередь: `operation`, `sourceEnvId`, `branch`, `status` (`pending`, `running`, …), временные метки, `requestedBy`;
  - журнал: связь с очередью или прямой запуск (`source`: `queue` | `script`), `output` / `error`, `durationMs`.
- Индексы по `projectId`, `status`, `createdAt`.

### Этап E3. API parity (project-scoped)

- `GET/POST/PATCH/DELETE` сред для проекта (расширить текущие роуты; **не** полагаться на глобально уникальное имя).
- `GET` статуса: агрегат по средам + хвост очереди + последние записи лога (аналог `deploy/status`).
- `POST` в очередь: постановка операции (deploy / copy между средами) с валидацией «нет параллельного running на ту же среду» — логика из `deploy/environments/[id]` komissionka.
- `POST deploy/copy` — перенос сценария копирования между средами с проверкой pending (см. komissionka `api/deploy/copy`).

### Этап E4. Worker и скрипты

- Адаптировать `deploy-worker.ts`: чтение из БД портала, вызов существующих shell-скриптов деплоя из **workspace целевого проекта** (параметризация через `DeployEnvironment.directory` / env vars).
- Документировать в `docs/` требования: права пользователя, пути к `deploy-from-git.sh` / аналогам, переменные окружения.
- Логирование каждого шага в `DeployLogEntry` (как в komissionka: источник `script` vs `queue`).

### Этап E5. UI в unified control panel

- `DeployEnvironmentsPanel`: карточки сред, порты, ветка, статус, prod-badge; кнопки «В очередь», «Копировать из…», просмотр последних логов и очереди.
- Единый UX с Backlog/Tests (фильтры, ошибки, polling статуса при длительных операциях).

### Этап E6. Миграция данных из komissionka

- Скрипт `scripts/migrate-deploy-from-komissionka.ts`:
  - перенос `deploy_environments` → записи `DeployEnvironment` с `projectId` проекта `komissionka`;
  - опционально перенос последних N записей `deploy_log` / открытых `deploy_queue` (с маппингом uuid сред).
- После миграции — переключение админки komissionka только на чтение или отключение дублирующих экранов (вне scope кода портала, зафиксировать в runbook).

## 7) Технические принципы (фиксируем)

- Без hardcode списка проектов (источник истины: workspaces + DB metadata).
- Любые shell-вызовы только через безопасный `spawn(args[])`, без shell interpolation.
- Все API под `adminAuthOk` до внедрения RBAC (п. 3.1); после — проверка прав проекта; для агентных интеграций можно расширить `x-agent-api-key`.
- Вызовы сторонних LLM — через сервис п. 3.2, ключи не прошивать во фронт.
- Ошибки возвращать структурированно:
  - `error`
  - `warnings[]`
  - `details` (если применимо)
- Не ломать текущие вкладки, мигрировать постепенно через shared-компоненты.

## 8) Рекомендуемая последовательность фаз (от простого к сложному)

### 8.1. Зачем такой порядок

1. **Бэклог первым** — минимальная цена входа (данные + API), сразу **dogfooding**: все хотелки по Shectory Portal (и остальной монолит) вести в проекте с slug вроде **`shectory-portal`** или **`cursorrpa`** в том же трекере, не в отдельных списках. Остальные фичи портала оформляются как задачи бэклога.
2. **UI панели — инкрементально:** сначала только бэклог в `/control`, чтобы не блокировать работу ожиданием тестов/деплоя.
3. **Generate Prompt (A3)** — после того, как в бэклоге уже есть реальные карточки и стабильные поля (A2).
4. **Тест-кейсы (D)** — до полного деплоя-worker: меньше инфраструктурного риска, API и схема изолированы.
5. **Деплой (E)** с **worker** и очередью — тяжелее (права, скрипты, долгие операции); логично после привычки к API и UI других блоков.
6. **Миграции данных из komissionka (A7, D6, E6)** — когда новый поток уже опробован на портале и komissionka; иначе легко упереться в двойное ведение.
7. **AI gateway (C)** — можно начинать с C0 под флагом после P1, если нужен учёт токенов для A3; полный C2/C3 и связка с RBAC — позже.
8. **RBAC (B)** — когда появляется реальная потребность в не-админах; до этого `adminAuthOk` не усложняет критический путь.

### 8.2. Фазы (единый «лестничный» план)

| Фаза | Что делаем | Этапы (ссылка на §4–6) | Сложность |
|------|------------|-------------------------|-----------|
| **P0** | Зафиксировать контракт бэклога и **slug проекта для хотелок портала** (одна запись `Project` в БД / реестре). | A0 | низкая |

### 8.3. Параллельные потоки (не ломая P1–P4)

- **После P1:** разработчик может вести ветку **C0** (внутренний `/api/ai/...`), не блокируя бэклог-UI.
- **После P3:** **D0** (аудит тестов) можно делать параллельно с P4, если A3 не занимает всю команду.
- **P6** не начинать до появления **минимального E3** на staging без worker (ручной прогон скриптов) — по желанию как подфаза P6a.

### 8.4. Соответствие старым «итерациям»

Ранее использовались названия «итерация 1…7»; они **заменены** таблицей P0–P10 выше, чтобы порядок отражал **зависимости и приоритет бэклога**, а не номера этапов A/D/E в документе.

## 9) Definition of Done для Control Panel

Считаем **ядро** (фазы **P1–P4**) состоявшимся, когда:

- Для проекта портала в бэклоге ведутся **реальные хотелки** по развитию Shectory Portal (dogfooding).
- Есть кнопка **«Панель управления»** и рабочий **бэклог** в `/control` (**P3**).
- Backlog по API и полям готов к parity с komissionka (**P1–P2**, далее **P7**/A7 для импорта истории).

Считаем задачу **полного** Control Panel завершенной, когда:

- Backlog/tests/deploy имеют единый UI-вид и единый UX-паттерн (**P5–P6** + оболочка **P3**).
- Backlog функционально покрывает parity с komissionka (минимум API + поля + статусный флоу; миграция **A7** в **P7**).
- Созданный проект (например PingMaster) проходит полный сценарий:
  - есть в реестре,
  - имеет workspace/git/readiness,
  - управляется из unified control panel.

**Тесты и деплой (после фаз P5–P6 и миграций P7):**

- Каталог тест-кейсов для проекта komissionka в портале **функционально сопоставим** с komissionka: поля, прогоны, запуск из UI; данные перенесены скриптом D6 или подтверждён план ручного импорта.
- Среды деплоя, очередь и журнал для komissionka доступны из панели; worker выполняет операции с записью в лог; миграция E6 выполнена или задокументирован обходной путь.

**Платформа (после фаз P8–P9):**

- Для выбранного проекта в панели можно назначить участников с ролями и ограничить доступ к функциям без обхода через глобальный админ-токен (кроме break-glass сценария, если оставлен).
- Вызовы внешних моделей для сценариев портала идут через общий AI-сервис; ключи не доступны с клиента; есть минимальный учёт/лимиты согласованно с политикой.
